# المزامنة الثنائية الاتجاه لأجهزة البصمة - المواصفات
# Bidirectional Fingerprint Device Sync - Specification

**الإصدار:** 1.0.0
**التاريخ:** 2026-07-20
**الحالة:** مسودة
**الوحدة:** `Modules/FingerprintDevices` + `Modules/AttendanceIntegration`
**Driver (سائق الجهاز):** `zkteco`، `hikvision`

---

## 1. نظرة عامة (Overview)

تتيح هذه الميزة **مزامنة ثنائية الاتجاه** بين تطبيق HRM وجهاز البصمة الفيزيائي عبر جسر Python موجود (`zkteco-service`):

| الاتجاه | الوصف | الحالة قبل الميزة |
|---------|------|-------------------|
| **من الجهاز إلى التطبيق** (Pull) | جلب المستخدمين/البصمات/الحضور/صور الوجه من الجهاز وتخزينها في قاعدة البيانات | **موجود جزئياً** عبر `DeviceFullSyncService` |
| **من التطبيق إلى الجهاز** (Push) | إرسال المستخدمين والبصمات من قاعدة البيانات إلى الجهاز الفيزيائي | **غير موجود** – جديد |

تدعم الميزة نوعَي أجهزة رئيسيين يختلفان في البروتوكول:

- **ZKTeco** (Time / iClock) — بروتوكول `pyzk` عبر `http://127.0.0.1:5000` (TCP/UDP على منفذ 4370).
- **Hikvision** (Access Control) — بروتوكول ISAPI + HTTP Digest عبر `http://127.0.0.1:5001` (HTTP/HTTPS).

> الهدف: أن يستطيع مدير النظام **نشر قائمة موظفين وبصماتهم على جهاز البصمة** بنفس سهولة جلب البيانات منه، مع تتبع النجاح/الفشل لكل سجل، ومع الحفاظ على نفس واجهة المستخدم (Vue) وطبقات Laravel الحالية (Controller → Service → Repository → Model).

## 2. قصص المستخدمين (User Stories)

- [ ] كـ **مدير نظام**، أستطيع دفع جميع المستخدمين من التطبيق إلى جهاز ZKTeco بنقرة واحدة مع متابعة التقدم.
- [ ] كـ **مدير نظام**، أستطيع دفع جميع المستخدمين من التطبيق إلى جهاز Hikvision بنقرة واحدة.
- [ ] كـ **مدير نظام**، أستطيع دفع بصمات موظف واحد (أو عدة موظفين مختارين) إلى جهاز البصمة.
- [ ] كـ **مدير نظام**، أستطيع تنفيذ مزامنة "مزدوجة" (سحب + دفع) في عملية واحدة بدلاً من عمليتين منفصلتين.
- [ ] كـ **مدير نظام**، أستطيع رؤية عدد السجلات التي نجح إرسالها وعدد التي فشلت، مع سبب الفشل.
- [ ] كـ **مدير نظام**، أستطيع الدفع إلى جميع الأجهزة النشطة دفعة واحدة.
- [ ] كـ **مدير نظام**، أستطيع الدفع الانتقائي: موظفين حسب الفرع، أو الموظفين الذين لا يملكون UID على الجهاز.
- [ ] كـ **مدير نظام**، عندما يفشل الدفع، يبقى الفشل معزولاً (لا يُلغي باقي السجلات).
- [ ] كـ **مدير نظام**، أستطيع رؤية آخر وقت دفع/سحب لكل جهاز في قائمة الأجهزة.

## 3. سيناريوهات الاستخدام (User Scenarios & Testing)

### السيناريو 1 — الدفع الكامل للموظفين إلى جهاز واحد
**الفاعل:** مدير النظام (صلاحية `edit-fingerprint-devices`).
**التمهيد:** يوجد 50 موظف في HRM، وجهاز ZKTeco متصل بالإنترنت.
**الخطوات:**
1. يفتح المدير صفحة `/fingerprint-devices/sync` ويختار الجهاز من القائمة.
2. يفعّل خيار "دفع المستخدمين إلى الجهاز" (`push_users`).
3. يفعّل خيار "دفع البصمات" (`push_fingerprints`).
4. يضغط "تشغيل المزامنة".
5. تظهر شاشة تقدم في الوقت الفعلي (SSE) تعرض:
   - المرحلة الحالية (users، fingerprints).
   - عدد المُرسَل بنجاح، عدد الفاشل.
   - السجلات الفاشلة مع سبب الفشل.
6. عند الانتهاء، يظهر ملخص + خيار إعادة محاولة السجلات الفاشلة فقط.

**معايير القبول:**
- ✅ تظهر رسالة "تم دفع 50 موظف، 48 بصمة، فشل 2 موظف" مع قائمة بالموظفين الفاشلين.
- ✅ الـ sync timestamp للجهاز يتحدث.
- ✅ صفحة الجهاز (`Show.vue`) تعرض آخر وقت دفع.

### السيناريو 2 — الدفع إلى جهاز Hikvision
**نفس السيناريو 1**، لكن الجهاز Hikvision (ISAPI + Digest Auth).
**معايير القبول الإضافية:**
- ✅ المصادقة تعمل عبر `username:password` (المخزّن في `comm_key` أو env).
- ✅ بصمة الوجه تُدفع إذا كانت متوفرة على الموظف (`face_photo_path`).

### السيناريو 3 — مزامنة مزدوجة (سحب + دفع) في عملية واحدة
**الخطوات:**
1. المدير يفعّل الخيارات: `info` + `users` (pull) + `push_users` + `push_fingerprints` + `attendance` (pull).
2. يضغط "تشغيل".
3. النظام ينفذ المراحل بالترتيب:
   - `info` → `pull_users` → `pull_fingerprints` → `pull_attendance` → `push_users` → `push_fingerprints`.
4. كل مرحلة تعرض تقدمها الخاص.
5. النتيجة النهائية تحتوي على مجاميع السحب والدفع.

### السيناريو 4 — الدفع الانتقائي لموظف واحد
**الخطوات:**
1. من صفحة المستخدم (User Show)، أو من قالب البصمات، يختار المدير "دفع إلى الأجهزة".
2. تظهر قائمة بالأجهزة المتاحة.
3. يختار جهازاً واحداً أو أكثر.
4. يضغط "دفع".

### السيناريو 5 — معالجة فشل جزئي
**التمهيد:** 100 موظف، منهم 3 بـ `employee_code` فارغ، والجهاز به منفذ خاطئ.
**التوقع:**
- الجهاز يُعلَّم `offline` بسبب فشل اختبار الاتصال.
- تظهر رسالة خطأ واضحة "الجهاز غير متاح"، لا يتم الدفع.
- السجل الفارغ من `employee_code` يُتجاهل مع تنبيه.

### السيناريو 6 — الدفع لجميع الأجهزة النشطة
**الخطوات:**
1. من قائمة الأجهزة (`/fingerprint-devices`)، يضغط "مزامنة الكل".
2. النظام يدفع المستخدمين/البصمات إلى كل جهاز `status=online`.
3. تظهر نتيجة مجمّعة.

## 7. نقاط التكامل (Integration Points)

### 7.1 الجسر Python (zkteco-service)

| المحول (Driver) | نقطة النهاية (Endpoint) | الاستخدام |
|----------------|----------------------|----------|
| `zkteco` | `POST /device/add-user` | دفع موظف واحد |
| `zkteco` | `POST /device/add-users-batch` | دفع دفعة |
| `zkteco` | `POST /device/export-template` | دفع بصمة |
| `zkteco` | `POST /device/export-templates-batch` | دفع دفعة بصمات |
| `zkteco` | `POST /device/get-users` | سحب قائمة المستخدمين |
| `hikvision` | `POST /device/add-user` | دفع موظف (مع face_url اختياري) |
| `hikvision` | `POST /device/export-template` | دفع بصمة |
| `hikvision` | `POST /device/get-face-photos` | سحب صور الوجه |

> **ملاحظة:** جميع endpoints موجودة في `zkteco-service/app.py` و `zkteco-service/hikvision_service.py`. الميزة الجديدة لا تحتاج إلى تعديل الجسر، فقط إلى Layer Laravel أعلى منه.

### 7.2 الطبقات (Architecture)

```
Vue (Sync.vue) 
   ↓ Inertia + SSE
Controller: DeviceFullSyncController
   ↓ يفوّض
Service: DeviceSyncOrchestrator (pull) + DevicePushService (push جديد)
   ↓ يحل driver
DeviceAdapterResolver → ZKTecoAdapter | HikvisionAdapter
   ↓ HTTP
zkteco-service (Python Flask)
   ↓ pyzk / ISAPI
Physical Device
```

## 8. الافتراضات (Assumptions)

1. **جسر Python قيد التشغيل** عند المستخدم (افتراضياً: `http://127.0.0.1:5000` و `http://127.0.0.1:5001`).
2. **كل جهاز له `deviceType.manufacturer`** يطابق `zkteco` أو `hikvision` (يحدد من `FingerprintDeviceTypeSeeder`).
3. **`employee_code` فريد لكل موظف** ويُستخدم كمعرّف على الجهاز.
4. **البصمات مخزّنة بصيغة base64** في `user_fingerprints.template_data` ومتاحة للدفع بعد جلبها من جهاز سابق.
5. **الجهاز يدعم إضافة مستخدمين عبر الشبكة** — في حال عدم الدعم (firmware قديم)، يُعرض تحذير.
6. **الشبكة بين الخادم والجهاز LAN** — لا يُتوقع دفع عبر WAN بسبب firewall.
7. **حجم البصمة ≤ 10KB** (قاعدة ZKTeco).
8. **عدد الموظفين لكل جهاز ≤ 3000** (حد ZKTeco).
9. **إعادة المحاولة (Retry) عند الفشل** تتم مرة واحدة فقط (لتجنّب الإغراق).
10. **القيم الافتراضية**: timeout=30s للدفع، 300s للسحب الجماعي، batch=50 سجل.

## 9. القيود (Out of Scope — خارج النطاق)

| البند | السبب |

## 10. المخاطر (Risks & Mitigations)

| المخاطرة | الاحتمال | التأثير | التخفيف |
|---------|---------|--------|--------|
| جسر Python غير متاح | متوسط | عالي | رسالة واضحة + فحص دوري للـ health + إعادة محاولة |
| firmware قديم لا يدعم إضافة مستخدمين | متوسط | عالي | اختبار مسبق (probe) قبل الدفع + تقرير بالأجهزة غير المتوافقة |
| بطء الدفع لـ 1000+ موظف | عالي | متوسط | Queue jobs + دفع دفعات |
| انقطاع الشبكة أثناء الدفع | متوسط | متوسط | كل سجل مستقل (no transaction)، إعادة المحاولة متاحة |
| تضارب UID على الجهاز | منخفض | عالي | قراءة قائمة UIDs الحالية قبل الدفع + استخدام `add_users_batch` الذكي |
| بصمة تالفة/غير متوافقة | متوسط | منخفض | تحقّق من base64 قبل الدفع + تخطّي بصمة مع تحذير |
