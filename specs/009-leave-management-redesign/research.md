## 1. بحث آلة حالة الطلبات (Leave Request State Machine)

### القرار
اعتماد نموذج سلسلة خطية (Linear Chain) مع حالات إضافية للإلغاء.

### المبررات
- يتوافق مع نموذج الموافقات الموجود في النظام
- يدعم 3 مستويات موافقة: مدير القسم → الموارد البشرية → المدير العام
- يسمح بإرجاع الطلبات للتعديل
- يدعم طلبات إلغاء الإجازات المعتمدة

### البدائل المدروسة

| البديل | المزايا | العيوب | السبب في رفضه |
|--------|---------|-------|---------------|
| سلسلة خطية | بسيط، سهل الفهم | غير مرني | - (مقبول) |
| قابل للتعديل من Admin | مرن جداً | معقد، صعب الصيانة | التعقيد غير مبرر |
|تخطي تلقائي | أسرع | يصعب تتبعه | يخالف المتطلبات |

### الحالات المقبولة

```
draft → submitted → manager_approved → hr_approved → gm_approved
                                                      ↓
                                                 pending_cancellation
                                                      ↓
                                                 cancelled
```

## 2. بحث حساب أرصدة الإجازات (Leave Balance Calculation)

### القرار
اعتماد حساب مبني على سنوات الخدمة (Tenure-based) مع إنشاء تلقائي للسنة الجديدة.

### جدول الرصيد

| سنوات الخدمة | الأيام السنوية |
|-------------|---------------|
| 1 - 5 سنوات | 15 يوم |
| 6 - 10 سنوات | 21 يوم |
| 11 - 15 سنة | 26 يوم |
| +15 سنة | 30 يوم |

### المبررات
- يتوافق مع المتطلبات الوظيفية FR-030
- يدعم إنشاء تلقائي للرصيد عند بداية السنة الجديدة
- يسمح بالتعديل اليدوي من Admin

### آليات الخصم والإعادة

| الحدث | العملية | الشرط |
|-------|---------|-------|
| موافقة نهائية | `used_days += days_count` | الحالة = `gm_approved` |
| إلغاء إجازة معتمدة | `used_days -= days_count` | الحالة = `cancelled` بعد موافقة الإلغاء |
| تعديل يدو | `total_days = value` | Admin فقط |

## 3. بحث نظام الإشعارات (Notification System)

### القرار
استخدام نظام إشعارات مزدوج: Socket.IO للإشعارات الفورية + قاعدة البيانات للإشعارات المحفوظة.

### المبررات
- Socket.IO موجود ومُهيأ في النظام
- يدعم الإشعارات الفورية للمستخدمين المتصلين
- قاعدة البيانات تحفظ الإشعارات للمستخدمين غير المتصلين

### أنواع الإشعارات

| النوع | الحدث |
|-------|-------|
| `info` | طلب جديد، تحديث حالة |
| `success` | موافقة، إتمام |
| `warning` | طلب تعديل |
| `error` | رفض، خطأ |

### هيكل الإشعار

```json
{
  "title": "طلب إجازة جديد",
  "message": "لديك طلب إجازة جديد من محمد الدوسري",
  "type": "info",
  "related_request_id": 42,
  "related_user_id": 5
}
```

## 4. بحث تخزين الملفات (File Storage)

### القرار
استخدام التخزين المحلي مع هيكل مسارات آمن.

### المبررات
- يتوافق مع النمط الموجود في النظام
- لا يحتاج تكامل مع خدمات سحابية
- يسمح بالتحكم الكامل في الصلاحيات

### مسارات التخزين

| النوع | المسار | الامتدادات |
|-------|--------|-----------|
| مرفقات الطلبات | `uploads/docs/` | PDF, PNG, JPG, DOCX |
| الصور الشخصية | `uploads/avatars/` | PNG, JPG |

### قواعد الحماية

- التحقق من امتداد الملف
- التحقق من نوع MIME
- الحد الأقصى للحجم: 10MB للمرفقات، 5MB للصور
- منع رفع ملفات executable

## 5. بحث تصدير PDF مع دعم العربي (PDF Export with Arabic)

### القرار
استخدام مكتبة تدعم إعادة تشكيل النص العربي مع خط Arial.

### المبررات
-Arial موجود على جميع الأنظمة
- arabic-reshaper يحل مشكلة تشكيل الحروف
- bidi يدعم الاتجاه من اليمين لليسار

### مسارات الخط المدعومة

| النظام | المسار |
|--------|--------|
| Windows | `C:/Windows/Fonts/arial.ttf` |
| Linux | `/usr/share/fonts/truetype/msttcorefonts/arial.ttf` |
| macOS | `/Library/Fonts/Arial.ttf` |

### آلية السقوط (Fallback)
تجربة مسارات متعددة حتى يتم العثور على الخط.

## 6. بحث نموذج الطبقات (Layered Architecture)

### القرار
تطبيق نموذج Controller → Service → Repository → Model.

### المبررات
- يتوافق مع مبادئ المشروع
- يفصل بين المسؤوليات
- يسهل الاختبار
- يدعم إعادة الاستخدام

### المسؤوليات

| الطبقة | المسؤولية | مثال |
|--------|----------|------|
| Controller | HTTP + Inertia::render() + redirect | `LeavesController` |
| Service | Business logic, validation, cache | `LeaveService` |
| Repository | Eloquent queries, filtering | `LeaveRepository` |
| Model | Relationships, scopes, attributes | `LeaveRequest` |

### القيود
- ❌ لا business logic في Controller
- ❌ لا DB queries في Controller
- ❌ لا app()/resolve() في Services - استخدام DI
- ✅ استخدام FormRequest للتحقق
- ✅ استخدام Resource لتحويل البيانات

## 7. بحث صلاحيات الوصول (Access Control)

### القرار
استخدام نظام الصلاحيات الموجود مع إضافة صلاحيات جديدة للإجازات.

### الصلاحيات المطلوبة

| الصلاحية | الوصف |
|----------|-------|
| `view-leave-requests` | عرض طلبات الإجازة |
| `create-leave-requests` | إنشاء طلبات إجازة |
| `approve-leave-requests` | الموافقة على طلبات إجازة |
| `manage-leave-balances` | إدارة أرصدة الإجازات |
| `export-leave-reports` | تصدير تقارير الإجازات |

### قواعد الوصول

| الدور | الصلاحيات |
|-------|----------|
| employee | إنشاء طلباته، عرض أرصدته |
| department_manager | الموافقة على طلبات قسمه |
| development_manager | عرض جميع الطلبات، الموافقة |
| general_manager | الموافقة النهائية |
| admin | كامل الصلاحيات |

## 8. بحث التكامل مع نظام الحضور (Attendance Integration)

### القرار
ربط طلبات الإجازة بسجلات الحضور عبر تحديث حالة الحضور.

### المبررات
- يمنع تداخل الإجازات مع المناوبات
- يحديث سجلات الحضور تلقائياً
- يدعم حساب أيام العمل الفعلية

### آلية التكامل

1. عند الموافقة على إجازة → تحديث `attendance_sessions`
2. تعيين حالة الحضور كـ `on_leave`
3. منع تسجيل حضور جديد أثناء فترة الإجازة

## 9. بحث التكامل مع نظام الرواتب (Payroll Integration)

### القرار
تمرير بيانات الإجازات المؤجرة لنظام الرواتب عبر API.

### المبررات
- يدعم حساب الرواتب الشهرية
- يميز بين الإجازات المؤجرة وغير المؤجرة
- يوفر بيانات دقيقة لنظام الرواتب

### البيانات الممررة

| الحقل | الوصف |
|-------|-------|
| `employee_id` | رقم الموظف |
| `leave_type` | نوع الإجازة |
| `days_count` | عدد الأيام |
| `is_paid` | هل هي مؤجرة |
| `period` | الفترة الزمنية |
